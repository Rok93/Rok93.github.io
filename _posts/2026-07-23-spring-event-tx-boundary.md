## 어떤 PR에서 출발했나

한 PR(코드 변경 제안)이 이런 변경을 했다. 어떤 데이터 한 건(이 글에선 "청구서"라고 부르겠다)이 **만들어질 때**가 아니라 **처음 발행될 때** "진행중" 상태가 되도록 순서를 바꾼 것이다. 회사 코드 대신 **결제/청구**라는 예시로 옮겨 설명한다.

- `Invoice`(청구서)를 만들 때 초기 상태를 `DRAFT`(초안)로 둔다. (예전엔 바로 `ISSUED`(발행됨)였다)
- 청구서를 처음 발행하면, "발행됐다"는 신호(이걸 **도메인 이벤트**라고 부른다 — "무슨 일이 벌어졌다"고 알리는 메시지)를 통해 상태를 `DRAFT → ISSUED`로 바꾼다.

리뷰에서 쟁점은 딱 두 가지였다. **이 뒷처리를 "이벤트"로 알려서 처리할까, 그냥 함수를 직접 부를까? 그리고 원래 작업과 같은 "트랜잭션" 안에서 처리할까, 따로 뗄까?** 이 한 PR을 파고드니 백엔드 핵심 개념이 줄줄이 엮여 나왔다. 하나씩 풀어보자.

## ① Spring 이벤트는 기본이 "동기(synchronous)"다

먼저 용어. **이벤트(event)**는 "이런 일이 생겼어!"라고 알리는 신호고, 그 신호를 받아 실행되는 함수를 **리스너(listener)**라고 한다. 신호를 쏘는 행위는 **발행(publish)**이라 부른다.

`ApplicationEventPublisher.publishEvent()`(이벤트를 쏘는 Spring 코드)는 **별도의 스레드(동시에 도는 다른 작업 흐름)에서 나중에** 실행되는 게 아니라, **바로 그 자리에서 즉시** 리스너를 실행한다. 이걸 **동기**라고 한다. "이벤트 = 나중에 따로 도는 것(비동기)"이라는 건 흔한 오해다.

그럼 그냥 함수를 직접 부르는 것과 뭐가 다른가? 이벤트를 쓸 이유는 딱 둘이다.

- **디커플링(느슨한 연결)**: 신호를 쏘는 쪽이 "누가 이걸 처리하는지" 몰라도 된다. 서로 직접 알지 않아 나중에 고치기 쉽다.
- **팬아웃(fan-out, 한 신호에 여러 반응)**: 한 번의 신호에 여러 리스너를 동시에 매달 수 있다.

이 둘이 필요 없다면 이벤트는 그냥 단계만 늘리는 **오버엔지니어링**(필요 이상으로 복잡하게 만든 것)이다. 그런데 이 경우엔 같은 이벤트에 리스너가 **2개**(상태 바꾸기 / 부수적인 플래그 갱신) 붙어 있었다. 그러니 팬아웃이라는 명분이 실제로 있었다.

## ② @EventListener vs @TransactionalEventListener = "언제" 실행되나

이벤트를 쓰기로 했다면, 다음 갈림길은 **리스너가 트랜잭션의 어느 시점에 실행되느냐**다.

잠깐, **트랜잭션(transaction)**부터. 여러 개의 DB 작업을 **"전부 성공 아니면 전부 취소"** 로 묶는 단위다. 확정해서 저장하는 걸 **커밋(commit)**, 도중에 문제가 생겨 통째로 되돌리는 걸 **롤백(rollback)**이라 한다.

이 코드가 하는 일: 상태 바꾸기 리스너(위)와 부수 플래그 리스너(아래)가 **각각 언제 도는지**를 애노테이션으로 지정한다.

```java
// 상태 바꾸기 — 같은 트랜잭션 안에서, 커밋 전에 즉시 실행. 실패하면 원래 작업까지 통째로 롤백된다.
@EventListener
public void syncStatusOnIssued(InvoiceIssuedEvent event) { ... }

// 부수 플래그 갱신 — 커밋이 끝난 뒤, 새 트랜잭션에서 실행
@TransactionalEventListener
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void markEmailed(InvoiceIssuedEvent event) {
    try {
        invoiceRepository.markEmailedFirstTimeByIds(event.invoiceIds());
    } catch (Exception e) {
        log.error("메일 발송 플래그 갱신 실패: {}", event.invoiceIds(), e);
    }
}
```

- `@EventListener` → 발행하는 즉시, **커밋 전에, 같은 트랜잭션 안에서** 실행된다. 실패하면 원래 작업까지 함께 롤백된다(이걸 **원자적**이라 한다 — 전부 되거나 전부 안 되거나).
- `@TransactionalEventListener` → 기본적으로 **커밋이 끝난 뒤**(AFTER_COMMIT) 실행된다. 여기에 `REQUIRES_NEW`(항상 새 트랜잭션을 열라는 옵션)를 붙이면 **완전히 별도의 트랜잭션**에서 돈다.

즉 **같은 이벤트인데 두 리스너의 처리 방식이 정반대**였다. 하나는 "같은 묶음 안, 커밋 전", 다른 하나는 "커밋 후, 별도 묶음". 처음엔 이 불일치를 문제(스멜)로 의심했다.

## ③ 같은 트랜잭션이냐 별도냐 = "일관성"을 어디까지 지킬 거냐

여기서 **일관성(consistency)**은 "여러 데이터가 서로 아귀가 맞는 정도"를 뜻한다. 두 갈래가 있다.

| | 강한 일관성 (같은 트랜잭션) | 최종 일관성 (별도/나중에 처리) |
|---|---|---|
| 보장 | 커밋되는 순간 항상 서로 일치 | 잠깐 어긋났다가 곧 맞춰짐 |
| 대가 | 응답이 느려지고, 한쪽 실패가 전체로 번짐 | 잠깐 안 맞는 "빈틈" 구간이 생겨 재처리가 필요 |
| 언제 쓰나 | 어긋나면 곧바로 잘못된 판단을 내리는 데이터 | 잠깐 어긋나도 큰일 안 나는 데이터 |

## ④ "load-bearing 상태"인지가 일관성 요구를 정한다

**load-bearing**은 건축에서 "하중을 받는(무너지면 큰일 나는) 기둥"을 뜻한다. 데이터에 빗대면 **"그 값을 다른 로직이 읽고, 그걸 근거로 중요한 판단을 내리는가"** 다.

판별 기준은 한 문장으로 요약된다. **"그 값을 읽고 곧바로 허용/거부를 결정하는 코드가 있는가?"**

이 코드가 하는 일: 청구서를 취소/수정해도 되는지를 `status`(상태) 값으로 판단한다. 즉 `status`가 판단의 근거가 된다.

```java
public boolean isEditable() {
    return status == Status.DRAFT || status == Status.ISSUED;
}

public void cancel(...) {
    if (!isEditable()) {
        throw new BusinessException("이 상태에서는 취소/수정할 수 없습니다.");
    }
    ...
}
```

- **status(상태)**: 위처럼 취소/수정 가능 여부를 이 값으로 **결정(게이팅 — 문을 열고 닫듯 허용/차단)**한다 → load-bearing이다 → 어긋나면 즉시 오판하므로 **강한 일관성(같은 트랜잭션)**이 필요하다.
- **isEmailed(부수 플래그)**: 목록 조회 필터·화면 표시·시각 기록용일 뿐, 이걸로 무언가를 허용/차단하지 않는다 → load-bearing이 아니다 → **최종 일관성**으로 충분하다.

## ⑤ "한 사건, 다른 처리 방식"은 문제일 수도, 정당할 수도

두 데이터의 **중요도(어긋났을 때의 위험)가 다르면**, 처리 방식이 다른 게 오히려 맞다.

- load-bearing 상태 → 발행과 **한 묶음(같은 트랜잭션)**으로 처리해 항상 아귀를 맞춘다.
- **best-effort**(되면 좋고, 실패해도 치명적이지 않은) 플래그 → 그 실패가 중요한 발행까지 되돌리지 않도록 **일부러 별도 트랜잭션으로 떼어 놓는다.**

원래 코드가 `REQUIRES_NEW`(새 트랜잭션) + 에러를 로그만 남기고 넘어가게 만든 건, "이 플래그는 best-effort다"라는 **의도적 설계**였던 것이다.

## ⑥ "자가복구"가 어느 값을 기준으로 도는지 확인하라

이 코드가 하는 일: 부수 플래그를 갱신하되, **아직 처리 안 된 건만** 골라 갱신한다.

```sql
update Invoice i
   set i.isEmailed = true, i.firstIssuedAt = current_timestamp
 where i.id in :ids and (i.isEmailed = false or i.isEmailed is null)
```

조건(`where`)이 `isEmailed = false`, 즉 **자기 자신**을 기준으로 한다. 그래서 첫 갱신이 실패해도, 다음에 다시 발행하면 이 쿼리가 또 돌아 **저절로 복구(self-heal)**된다. 나는 이걸 상태 리스너의 조건(`status != DRAFT면 아무 것도 안 함`)과 헷갈려 "한 번 실패하면 복구 불가"라고 단정했다가, 코드를 다시 보고 정정했다.

## 코드로 검증한 과정 (이 글의 핵심)

"부수 플래그가 정말 안 중요한가?"를 머리로 추측하지 않고, **그 값을 쓰는 곳을 전부 검색**했다. 읽는 곳은 목록 표시 / 검색·다운로드 필터 / 시각 기록뿐, **그 값으로 무언가를 허용/차단하는 곳은 없었다.** 이 한 번의 확인으로 두 가지가 뒤집혔다.

1. "처리 방식 불일치 = 문제" → **"중요도에 맞춘 정당한 설계"**
2. "실패하면 복구 불가" → **"다시 시도하면 저절로 복구"** (남는 건 '최초 시각'이 재시도 시각으로 살짝 어긋날 여지 정도)

결론적으로 **꼭 고쳐야 할 결함은 없었다.** 리뷰의 진짜 소득은 버그를 찾은 게 아니라, 왜 이 설계가 맞는지 **근거를 들어 설명할 수 있게 된 것**이었다.

## 한 줄 교훈

> 어떤 값이 중요한지(load-bearing인지)는 머리로 상상하지 말고, "누가 그 값을 읽고 무슨 결정을 하는지"를 코드에서 직접 확인해 정하라. 그게 이벤트/트랜잭션 설계를 리뷰하는 출발점이다.
